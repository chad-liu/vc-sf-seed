## 專案簡介

「三花菁英種子學堂」學校管理網站。學校以「縣市 → 學校簡稱」連動下拉選單加密碼登入，
登入後在 Dashboard 五個分頁操作：公告及注意事項、學校資料編輯、申請作業、上學期成果、下學期成果
（最後兩個分頁尚未實作）。規格來源：`plan.md`（整體）、`plan2.md`（申請作業分頁）、`學校基本資料.png`（畫面參考）。

## 環境變數（.env.local，不入版控）

```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=   # API routes 用此繞過 RLS
SESSION_SECRET=              # iron-session 加密金鑰（32+ 字元）
```

Supabase 實例與 D:\VC-SF-Story 專案**共用**（本專案用 `city`、`sf_school`、`sf_news`、
`sf_apply`、`sf_apply_detail` 五張表與 `sf_plan_pdf` bucket）。
**API routes 一律透過 `src/lib/supabase.ts` 的 client（service role key）存取**，不要另建 anon key client。

## 架構

### 認證流程
`src/lib/session.ts` 定義 `SessionData`（id、school、schoolNo、city）與 `getSession()`
（iron-session v8，async cookie API）。登入比對 `sf_school` 的 city + school + password
（明文，資料庫既有設計），密碼錯誤固定回「密碼輸入錯誤」。所有 API routes 先呼叫
`getSession()` 驗證，資料操作一律以 `session.schoolNo` 過濾，確保只能動自己學校的資料。

### Dashboard
`src/app/dashboard/page.tsx`（Server Component，未登入 redirect 回 `/`）→
`DashboardClient.tsx`（Client Component，分頁切換）。各分頁內容元件在 `src/components/`。

### 申請作業（ApplyForm.tsx + /api/apply*）
- `GET /api/apply`：該校在 `sf_apply` 無資料時**自動新增一筆**再回傳（year 由 DB 預設 '115'）。
- 申請資料與經費 grid 都採「預設唯讀、按修改進入編輯」模式；經費 grid 一次只允許一列編輯，
  「新增」會立即 POST 建立資料列並直接進入編輯模式。
- **編輯模式切換的按鈕一律用 `type="button"`**：同一位置的按鈕在點擊中途換成
  type="submit" 會觸發表單送出（曾造成「一閃而過」bug），故本專案不用 form submit 機制。

### PDF 上傳（/api/apply/upload）
Supabase Storage 物件名稱**不允許中文**（InvalidKey），故實體檔名為
`plan-{year}-{schoolno}.pdf`（upsert 覆蓋，舊命名檔案自動刪除），使用者看到的中文檔名
`課程規劃-{year}-{schoolno}-{school}.pdf` 靠下載 URL 的 `?download=` 參數
（Content-Disposition）呈現。公開 URL 有 CDN 快取（3600 秒），前端連結必須帶 `?t=` 時間戳。

### 資料表注意事項
- `sf_apply_detail`：整數主鍵 `id` 需為 identity（自動編號）。若「新增」回
  id not-null 錯誤，請使用者在 Supabase SQL Editor 執行：
  `alter table sf_apply_detail alter column id add generated by default as identity;`
- `sf_school.password` 為明文且會顯示於畫面（沿用參考畫面設計）。
- 資料表無法從程式端做 DDL；schema 變更都要請使用者在 Supabase Dashboard 執行。
